Quest Engine — PROJECT_NOTES.md
Загальний опис
Quest Engine — веб-платформа для проведення командних онлайн-квестів та змагальних ігор.
Платформа підтримує різні режими гри, різні типи завдань, бонуси, штрафи, автопереходи, адміністрування запусків та публічну статистику.

Backend:

•	Express
•	SQLite
Frontend:
•	HTML
•	CSS
•	Vanilla JavaScript

Режими гри
LINEAR
Класичний режим.
Команда бачить лише одне поточне завдання.
Після завершення поточного завдання відкривається наступне.
 
Переможець визначається за мінімальним ігровим часом.
Ігровий час:
Повний час
•	бонуси
•	штрафи
 
STORM
Режим паралельного виконання завдань.
Команда може мати відкритими одночасно багато завдань.
 
Можливі варіанти відкриття завдань:
Усі одразу
Після старту гри команда бачить усі завдання.
 
Групами
Приклад:
Після старту доступні:
1, 2, 3, 4
Через 3 години відкриваються:
5, 6, 7
 
Залежні завдання
Приклад:
Після виконання завдання 4 відкривається завдання 8.
 
Гра типу STORM завжди має заздалегідь визначений час завершення.
Після настання часу завершення гра автоматично закінчується.
 
Переможець визначається не за часом, а за балами.
Бали можуть нараховуватись:
•	за виконання завдань;
•	за бонуси;
•	за штрафи;
•	за інші механіки конкретної гри.
 
Якщо кілька команд мають однакову кількість балів:
вище розташовується команда, яка набрала цей результат раніше.
Час використовується як тайбрейкер.
 
Типи завдань
CLASSIC
Стандартне завдання з відповідями.
 
OLYMPIC
Планується до реалізації.
Потребує окремого проєктування.
 
Типи коригувань часу
BONUS_CODE
Бонусний код.
 
PENALTY_CODE
Штрафний код.
 
MANUAL_BONUS
Ручний бонус адміністратора.
 
MANUAL_PENALTY
Ручний штраф адміністратора.
 
AUTO_TRANSITION
Використовується для:
•	штрафів за автоперехід;
•	коригувань часу при примусовому завершенні завдань;
•	коригувань часу при групових переведеннях команд.
 
Поточна логіка статистики
Повний час
Сума часу проходження всіх завдань.
 
Ігровий час
Повний час
•	бонуси
•	штрафи
 
Рейтинг
Для LINEAR:
Переможець визначається за мінімальним ігровим часом.
Для STORM:
Переможець визначається за максимальною кількістю балів.

Поточний функціонал
Таблиця містить:
•	команда;
•	старт;
•	усі завдання;
•	фініш;
•	повний час;
•	бонуси та штрафи;
•	ігровий час;
•	відставання;

Виділення клітинок
При виборі клітинки:
•	виділяється рядок команди;
•	виділяється стовпець;
•	виділяється активна клітинка.
 
Деталі завдання
При натисканні на клітинку завдання показуються:
•	контент;
•	підказки;
•	відповіді;
•	знайдені відповіді;
•	лог відповідей;
•	бонуси та штрафи завдання.
 
Бонуси та штрафи
При натисканні на колонку бонусів/штрафів:
показуються всі коригування команди.
Доступно:
•	додати ручний бонус;
•	додати ручний штраф;
•	видалити ручний бонус;
•	видалити ручний штраф.
Видалення виконується через підтвердження.
 
Примусове завершення завдання
Доступне для завдання зі статусом:
“в процесі”.
 
Якщо автоперехід не налаштований:
•	completed_at = поточний час;
•	штрафи не додаються.
 
Якщо автоперехід налаштований:
Час завершення:
completed_at = поточний час.
 
Створюється коригування:
Час автопереходу мінус фактичний час перебування на завданні.
Коментар:
“Коригування за примусове завершення завдання №X”
 
Якщо був налаштований штраф за автоперехід:
додається окремий запис AUTO_TRANSITION.
 
Після завершення:
•	відкривається наступне завдання;
або
•	команда завершує гру.
 
Групові дії адміністратора
Реалізовано.
 
Адміністратор бачить список усіх команд запуску.
Поряд із кожною командою є чекбокс.
Доступно:
•	вибрати всі команди;
•	зняти вибір з усіх команд.
 
Адміністратор може перевести вибрані команди на будь-яке завдання.
Завдання обирається зі списку.
Перед виконанням дії показується підтвердження.
 
Для поточного завдання виконується та сама логіка, що і при примусовому завершенні завдання однієї команди.
 
Для всіх пропущених завдань:
•	час проходження встановлюється 00:00:00;
•	застосовуються коригування автопереходів;
•	застосовуються штрафи автопереходів.
 
Якщо команда вже пройшла завдання, на яке виконується переведення:
така команда пропускається і не змінюється.
 
Публічна статистика
Після завершення гри адміністратор може опублікувати результати.
 
Публікація:
•	переводить запуск в ARCHIVED;
•	генерує публічне посилання;
•	копіює посилання в буфер обміну;
•	показує посилання адміністратору.
 
Публічна статистика повинна підтримувати два режими:
Детальна
З часом проходження кожного завдання.
 
Стисла
Без деталізації по завданнях.
 
Найближчий план
1.	Групові дії адміністратора.
2.	Користувацька статистика.
3.	Архів запусків.
4.	Тип завдання OLYMPIC.
5.	Завантаження медіафайлів.
6.	Розвиток режиму STORM.
 
Важливо
Не змінювати бізнес-логіку без погодження.
Не змінювати структуру БД без погодження.
Перед великими змінами спочатку аналізувати поточний код і пропонувати план реалізації.

Користувацька статистика
Реалізовано першу публічну версію статистики для учасників.

Публічна сторінка показує:
• назву запуску;
• назву гри;
• дату й час старту гри;
• таблицю результатів без адмінських кнопок і службових деталей.

На публічній сторінці доступні два режими:
• Детально;
• Стисло.

Режим “Детально” показує:
• команда;
• старт команди;
• час проходження кожного завдання;
• час від старту до фінішу;
• бонуси та штрафи;
• ігровий час;
• відставання від попередньої команди;
• відставання від першої команди;
• місце.

Режим “Стисло” показує:
• команда;
• ігровий час;
• відставання від попередньої команди;
• відставання від першої команди;
• місце.

У публічній статистиці не показуються:
• час фінішу запуску над таблицею;
• окремий стовпець фінішу команди;
• технічний пояснювальний текст;
• час початку і завершення всередині клітинок завдань.

Доопрацювання завершення гри та публікації статистики
Реалізовано поведінку завершення гри для адміністративних дій.

Зміни:
• якщо команда завершена адміністратором або запуск архівований, гравець бачить фінальний екран гри;
• при публікації статистики система перевіряє, чи всі команди завершили гру;
• якщо є незавершені команди, адміністратор бачить попередження з кнопками “Скасувати” і “Усе одно завершити”;
• при підтвердженні незавершені команди автоматично переводяться на фініш;
• для пропущених завдань застосовуються коригування автопереходів і штрафи автопереходів;
• після публікації публічне посилання копіюється в буфер обміну;
• поряд із посиланням коротко показується повідомлення “скопійовано в буфер обміну”;
• у групових діях адміністратора додано пункт “Фініш”.

## Оновлення: режим перегляду архівної статистики

- В адмінській статистиці прибрано кнопку “← Назад”, оскільки статистика відкривається в окремій вкладці.
- Якщо запуск уже архівний, блок “Групові дії адміністратора” не показується.
- Якщо запуск уже архівний, чекбокси вибору команд у таблиці статистики не показуються.
- Архівна адмінська статистика працює як режим перегляду без елементів, що можуть змінити результати.

## Поточна зміна: вирівнювання вигляду статистики

- В адмінській статистиці заголовок першої колонки вирівняно так само, як інші заголовки.
- Додано проміжок між блоком публічного посилання та таблицею статистики.
- У публічній статистиці заголовок першої колонки відцентровано, при цьому назви команд у рядках залишаються вирівняними для читання.

## 2026-06-16 — Нормалізація часу UTC / локальний час користувача

### Реалізовано
- Додано спільний клієнтський модуль `client/time-utils.js` для роботи з датами й часом.
- Час, який адміністратор вводить у формах запуску, конвертується з локального часового поясу браузера в UTC перед збереженням у базу.
- Час у базі зберігається у форматі UTC `YYYY-MM-DD HH:MM:SS`.
- При виведенні часу на екран браузер конвертує UTC у локальний часовий пояс користувача.
- Оновлено створення запуску, редагування запуску, актуальні запуски, архіви, адмінстатистику, публічну статистику та екран очікування гри.
- Розрахунки тривалості й таймери продовжують працювати через UTC-час, щоб уникнути помилок через часові пояси.

### Правило проєкту
- У базі даних будь-який час зберігається тільки в UTC+0.
- Користувач вводить час у своєму локальному часовому поясі.
- Усі інтерфейси показують час у локальному часовому поясі поточного користувача.

- Сторінка `START` тепер рендериться гравцю, якщо він відкрив реєстраційне посилання або командне посилання до старту гри.
- Сторінка `FINISH` тепер рендериться гравцю після завершення гри або для архівного запуску.
- Сторінки `START`, `TASK` і `FINISH` залишаються повноцінними кастомними сторінками конструктора.
- Сторінка `JOIN` / «Реєстрація команди» більше не є контентною сторінкою конструктора.
- Для `JOIN` у редакторі гри залишено тільки налаштування системної сторінки: заголовок, опис, текст кнопки і CSS.
- Поле назви команди, кнопка створення команди, посилання на гру і кнопка копіювання залишаються системними елементами, щоб адміністратор не міг випадково зламати реєстрацію.
- API `/api/join/:runCode` тепер повертає статус `WAITING` або `READY` і відповідну сторінку гри.
- API `/api/play/:pin` тепер повертає кастомну сторінку `START` для очікування старту і `FINISH` для завершеної гри.
