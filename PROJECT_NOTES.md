

## FEATURE-005 — Профіль користувача і відновлення пароля

### Прийнята модель доступу

Відкрита реєстрація користувачів не планується. Для Quest Engine використовується закрита модель:

1. Адміністратор створює користувача в майбутньому розділі `Користувачі і ролі`.
2. Адміністратор передає користувачу тимчасові дані для входу.
3. Користувач входить у систему.
4. У меню `Редагувати профіль` користувач змінює логін, пароль і додає email для відновлення.

Це краще відповідає логіці закритої адмін-панелі рушія і зменшує ризик випадкових або сторонніх реєстрацій.

### Реалізовано

- Сторінка `profile.html` більше не є заглушкою.
- Користувач може змінити:
  - імʼя;
  - логін;
  - email для відновлення пароля;
  - пароль.
- Профіль і пароль зберігаються однією кнопкою `Зберегти зміни`.
- Для будь-якої зміни потрібен поточний пароль.
- Якщо новий пароль не вказано, змінюються тільки дані профілю.
- Якщо новий пароль вказано, після збереження сесія завершується і користувач має увійти повторно.
- Login-форма приймає логін або старий email.
- Додано сторінки `forgot-password.html` і `reset-password.html`.
- Додано серверні API:
  - `GET /api/auth/profile`;
  - `PUT /api/auth/profile`;
  - `GET /api/auth/mail-status`;
  - `POST /api/auth/forgot-password`;
  - `POST /api/auth/reset-password`.
- Додано SMTP-підготовку через `nodemailer`.
- Додано `.env.example` зі змінними для production-хостингу.

### Email-відновлення пароля

Для production потрібно заповнити змінні середовища:

- `APP_BASE_URL` — публічна адреса рушія, наприклад `https://quest.example.com`;
- `SESSION_SECRET` — довгий випадковий секрет сесії;
- `SMTP_HOST`;
- `SMTP_PORT`;
- `SMTP_SECURE`;
- `SMTP_USER`;
- `SMTP_PASS`;
- `MAIL_FROM`.

Якщо SMTP налаштовано, reset-link надсилається на `recovery_email`. Якщо SMTP не налаштовано, у dev-режимі reset-link показується на сторінці та дублюється в консолі сервера.

### Наступний логічний етап

Розділ `Користувачі і ролі`:

- створення нового користувача адміністратором;
- призначення ролі `ADMIN`, `AUTHOR`, `MODERATOR`;
- генерація тимчасового пароля;
- можливість вимагати зміну пароля після першого входу;
- передача/делегування прав на конкретні ігри.

# Quest Engine — PROJECT NOTES

## Поточний фокус

Розвиток конструктора сторінок гри після впровадження Tiptap WYSIWYG у текстовий блок.

## Реалізовано в поточному етапі

### Media sizing

Для окремих блоків `Зображення` і `Відео` додано налаштування розміру відображення:

- ширина;
- одиниця ширини: `px` або `%`;
- висота;
- одиниця висоти: `px` або `%`.

Якщо висота не вказана, вона розраховується автоматично з урахуванням пропорцій.

Для зображень у WYSIWYG-блоці при вставці через URL також запитується розмір зображення.

### Списки у WYSIWYG

Додано правило, щоб маркер або номер списку рухався разом із текстом при вирівнюванні по центру або правому краю.

### Секції / колонки

Додано новий тип блоку:

- `Секція / колонки`.

Секція дозволяє розміщувати контент не тільки зверху вниз, а й у ряд.

Налаштування секції:

- кількість колонок: від 1 до 4;
- ширина кожної колонки у відсотках;
- сума ширин колонок має дорівнювати 100%.

У кожну колонку можна додавати:

- Текст / WYSIWYG;
- Зображення;
- Відео;
- Аудіо;
- HTML/CSS/JS.

## Найближчі наступні кроки

- Перевірити стабільність секцій у START / TASK / FINISH сторінках.
- Покращити UX редагування колонок.

## Виправлення секцій

- Блок SECTION дозволений у серверній валідації контенту.
- Вкладені блоки в колонках секції проходять рекурсивну перевірку типів.
- Сума ширин колонок секції має дорівнювати 100%.

## Модель конструктора контенту

Верхній рівень сторінки складається із секцій.

- Секція є основним контейнером контенту.
- Нова секція за замовчуванням має одну колонку шириною 100%.
- Текст, зображення, відео, аудіо та HTML/CSS/JS додаються тільки всередину колонок секції.
- Кнопка «Додати секцію» розташована під уже доданим контентом.
- Якщо контенту немає, показується повідомлення «Тут ще немає жодного контенту».

## Тип завдання OLYMPIAD

Додано першу реалізацію типу завдання `OLYMPIAD`.

### Основна модель

Автор задає:

- кількість асоціацій `X`;
- кількість рівнів `L`.

Кількість клітинок розраховується автоматично:

```text
1 + X + X² + ... + X^(L-1)
```

Наприклад:

```text
X = 3
L = 4
1 + 3 + 9 + 27 = 40 клітинок
```

### Логіка проходження

- Коди можна вводити у будь-якому порядку.
- Коли команда здає правильний код, відкривається відповідна клітинка таблиці.
- У відкритій клітинці питання замінюється на відповідь.
- Логіка може працювати як знизу вгору, так і зверху вниз.

### Умови завершення

Підтримуються режими:

- здати верхню клітинку;
- здати всі коди;
- здати N будь-яких кодів.

### Блок «Відповіді»

Для `OLYMPIAD` додано налаштування приховування блоку «Відповіді» для гравця.
За замовчуванням для олімпійки цей блок має бути прихований.

## OLYMPIAD v2 — уточнена модель

Для типу завдання `OLYMPIAD` редактор має окрему логіку.

### Налаштування

- Кількість асоціацій `X`.
- Кількість рівнів `L`.
- Умова завершення:
  - здати верхню клітинку;
  - здати всі коди;
  - здати N будь-яких кодів.
- Поле `N кодів для завершення` показується лише для режиму «здати N будь-яких кодів».
- Можливість купівлі кодів через N секунд.
- Приховування блоку «Відповіді» для гравців.

### Клітинки

Для клітинок нижнього рівня редагуються:

- питання / вміст клітинки;
- відповідь;
- купівля.

Для клітинок верхніх рівнів редагуються:

- відповідь;
- купівля.

Поле «Купівля» у режимі `TIME` означає штраф у секундах. Для гравця кнопка показується як `Купити (+60 сек)`.
Поле «Купівля» у режимі `POINTS` означає штраф у балах. Для гравця кнопка показується як `Купити (-5 балів)`.

### Відображення для гравця

- Закриті клітинки нижнього рівня показують масштабований вміст клітинки.
- Вміст не обрізається, а зменшується до доступного розміру превʼю.
- Клік по превʼю відкриває повний перегляд у поп-апі.
- Відкриті клітинки показують відповідь.
- Якщо код здано звичайним способом, кнопка купівлі зникає і штраф не нараховується.
- Якщо код куплено, клітинка відкривається, код вважається зданим, а команді нараховується штраф.

## OLYMPIAD — виправлення після тестування

- Для завдань типу OLYMPIAD стандартний блок додавання кодів у вкладці кодів не використовується.
- Поле N кодів для завершення видиме тільки для режиму завершення COUNT.
- Клік по превʼю клітинки відкриває попап повного перегляду; попап не має закриватися через перемальовування таймера купівлі.
- Купівля коду має відкривати відповідну клітинку, показувати повідомлення про прийняття коду та враховуватися як штрафне коригування в статистиці.
- Якщо умовою завершення є верхня клітинка, купівля верхнього коду також завершує завдання.

## Поточне уточнення OLYMPIAD

- Нижні клітинки олімпійки мають бути компактними: контент масштабується до меж 200×100 px, але не додає зайвого порожнього простору.
- Клік по масштабованому вмісту відкриває повний перегляд; окрема кнопка збільшення не використовується.
- Блок налаштувань олімпійки має відображати кожне налаштування в окремому рядку; поле N показується лише для умови «здати N будь-яких кодів».

## Додаткове правило перевірки кодів

У редакторі завдань система перевіряє дублікати кодів перед збереженням. Якщо два коди однакові, показується alert з номерами кодів, наприклад: `Код 1 і код 8 однакові. Потрібно виправити.` Сторінка не перезавантажується, щоб автор не втратив заповнені дані.

# MULTITASK

Тип завдання `MULTITASK` призначений для випадків, коли в межах одного завдання є кілька незалежних підзавдань. Кожне підзавдання має власний контент, один основний код, опис формату відповіді, коментар і окрему підказку.

## Редактор

Для `MULTITASK` використовуються вкладки:

- Multitask
- Автоперехід

Вкладки Контент, Коди і Підказки приховуються.

У вкладці Multitask є:

- умова проходження: всі коди або N кодів;
- кнопка створення підзавдання;
- список підзавдань;
- кнопка створення підзавдання завжди внизу.

## Підзавдання

Підзавдання містить:

- контент через конструктор секцій;
- значення коду;
- опис, який показується гравцю як формат відповіді;
- коментар, який показується після прийняття коду;
- тип підказки.

## Підказки

Типи підказок:

- без підказки;
- підказка за часом;
- платна підказка.

У MULTITASK купується саме підказка, а не код. Купівля підказки не зараховує код автоматично.

## Player-flow

Команда бачить усі підзавдання в межах одного екрана. Коли код підзавдання прийнято, весь блок цього підзавдання замінюється зеленим рядком із номером і значенням прийнятого коду.

## MULTITASK — поточні уточнення після тестування

- Поле `N` для проходження має бути видимим тільки якщо обрано умову «Потрібно здати N кодів».
- Для типу підказки `Без підказки` додаткові поля не показуються.
- Для типу `Підказка за часом` показуються тільки текст підказки та час появи.
- Для типу `Платна підказка` показуються текст підказки, час появи кнопки купівлі та ціна купівлі.
- Окрема кнопка «Зберегти Multitask» не потрібна — усе зберігається загальною кнопкою «Зберегти всі зміни на сторінці».
- У редакторі завдань і сторінок гри блок дій із кнопками попереднього перегляду та збереження має бути закріплений внизу екрана.
- Групові дії адміністратора мають працювати по точному `team_id`, без помилок через схожі назви команд.


## POINTS MVP / standard hints update

- Додано базову підтримку режиму визначення переможця POINTS.
- Для завдань у POINTS додано поле нарахування балів за виконання завдання.
- Додано журнал points-подій із часом нарахування/списання балів.
- У статистиці POINTS рейтинг сортується за балами, а при рівності — за часом досягнення рахунку.
- У адмінстатистиці клітинка загального рахунку відкриває лог балів команди.
- Для STANDARD уніфіковано підказки: без підказки, підказка за часом, платна підказка.
- Для POINTS бонуси/штрафи, купівля підказок, купівля кодів олімпійки та автоперехід працюють у балах.
- Зафіксовано відкладене виправлення: preview має відповідати ширині реальної гри.

## POINTS — уточнення

- У бальному режимі бонуси, штрафи, купівля підказок і купівля кодів відображаються в балах.
- У STANDARD-підказках після створення підказки доступні лише “Підказка за часом” і “Платна підказка”; “Без підказки” не використовується, бо відсутність підказок уже означає цей стан.
- У OLYMPIAD попап відкривається лише для масштабованого вмісту клітинки.
- Купівля коду OLYMPIAD у TIME-режимі є автоматичним штрафом і не має видалятися як ручне коригування.

## Поточні уточнення POINTS

- POINTS є режимом визначення переможця, а не типом завдання.
- Основні коди STANDARD і MULTITASK не мають власної вартості; бали нараховуються за виконання завдання загалом.
- У POINTS статистиці рейтинг сортується за кількістю балів за спаданням, а при рівності — за меншим ігровим часом.
- Ігровий час у POINTS = час останнього нарахування балів мінус час старту команди.
- Всі події, що змінюють рахунок, мають timestamp.
- У каталозі ігор можна створити копію гри з новою механікою LINEAR/STORM і новою умовою перемоги TIME/POINTS.
- Якщо умова перемоги змінюється при копіюванні, поля секунд/балів у копії очищаються.
- Preview у редакторі має відображати завдання з обмеженнями ширини, близькими до player-flow.

## Поточні уточнення POINTS

- Поле «Нарахувати за виконання завдання ... балів» показується тільки для ігор з умовою перемоги POINTS.
- Для STANDARD підказок поля залежать від типу підказки: для підказки за часом показується тільки час появи, для платної підказки — час появи кнопки купівлі та вартість купівлі.
- У статистиці POINTS окрема колонка «Бонуси / штрафи» не використовується, бо всі події рахунку доступні в деталізації балів.
- Клітинки завдань у адмінстатистиці POINTS клікабельні й показують лог балів саме за вибране завдання.
- Публічна статистика POINTS має детальний і стислий режими та сортує команди за балами, а при рівності — за меншим ігровим часом.

## FEATURE-002 — порядок завдань і копіювання одного завдання

У редакторі гри додано керування конкретним завданням:

- кнопка `↑ Вище` переміщує завдання на одну позицію раніше;
- кнопка `↓ Нижче` переміщує завдання на одну позицію пізніше;
- після зміни порядку сервер перенумеровує `sort_order` усіх завдань гри з 1;
- активна вкладка завдання зберігається після переміщення.

Додано копіювання конкретного завдання в обрану гру:

- у редакторі завдання є список доступних ігор;
- кнопка `Скопіювати завдання` створює копію в кінці списку завдань цільової гри;
- якщо копіювання відбувається в ту саму гру, назва отримує префікс `Копія —`;
- копіюються базові поля завдання, контент, коди, підказки, OLYMPIAD-налаштування/клітинки та MULTITASK-підзавдання;
- звʼязки `task_answer_id` для OLYMPIAD і MULTITASK перемаплюються на нові коди;
- залежність STORM за іншим завданням (`unlock_type = TASK`) при копіюванні одного завдання скидається в `IMMEDIATE`, щоб не посилатися на завдання з іншої гри;
- якщо копіювання відбувається між іграми з різною умовою перемоги TIME/POINTS, часові/бальні модифікатори очищаються аналогічно копіюванню всієї гри.

## FEATURE-003 — Оптимізація редактора завдань

У редакторі завдань додано UX-покращення для безпечнішої роботи з великими завданнями:

- локальне автозбереження чернетки в браузері;
- індикатор стану чернетки поруч із назвою завдання;
- банер відновлення, якщо редактор відкрито повторно і знайдено локальну чернетку;
- попередження при закритті сторінки або переході між вкладками гри, якщо є незбережена чернетка;
- після успішного збереження на сервері локальна чернетка очищається;
- ущільнено верхній блок налаштувань, таблицю кодів, блоки підказок, поля OLYMPIAD і MULTITASK.

Автозбереження є локальною чернеткою, а не серверним збереженням: воно захищає від випадкового закриття вкладки або перезавантаження, але фінальне застосування змін виконується кнопкою `Зберегти всі зміни на сторінці`.

## FEATURE-004 — Попередня реєстрація команд

Для запуску гри додано додатковий режим реєстрації команд до старту гри.

Базовий режим за замовчуванням не змінюється:

- якщо запуск ще не стартував, посилання для реєстрації показує START/брифінг-сторінку і таймер до старту;
- після старту сторінка оновлюється і відкриває форму реєстрації команди.

Додатковий режим вмикається окремим чекбоксом у створенні та редагуванні запуску:

- чекбокс за замовчуванням вимкнений;
- назва чекбокса: `Відкрити попередню реєстрацію команд на гру`;
- якщо чекбокс увімкнено, посилання реєстрації команд одразу відкриває форму створення команди навіть до старту гри;
- після створення команди користувач отримує командне ігрове посилання;
- якщо перейти за командним ігровим посиланням до старту, учасники бачать START/брифінг-сторінку і таймер;
- якщо перейти за командним ігровим посиланням після старту, учасники одразу потрапляють у гру.

## PROFILE/AUTH — уточнення після тесту

Після зміни пароля користувача не варто залишати в старій активній сесії. Поточна логіка:

- зміна логіна/email відновлення оновлює поточну сесію без виходу;
- зміна пароля оновлює `password_hash`, очищає reset-токени і завершує поточну сесію;
- після зміни пароля користувач має зайти повторно з новим паролем;
- форми профілю мають блокувати повторне натискання кнопки під час запиту.

## FEATURE-006 — Користувачі, ролі і доступи до ігор

### Прийнята модель

Відкрита реєстрація користувачів не використовується. Користувачів створює адміністратор у розділі `Користувачі і ролі`, після чого передає тимчасові дані входу.

### Ролі

- `ADMIN` — повний доступ до всього рушія.
- `AUTHOR` — створює власні ігри, бачить власні ігри та ігри, до яких йому надали доступ.

Роль `MODERATOR` поки не використовується і прибрана з інтерфейсу.

### Доступи до ігор

Для кожної гри використовується власник `created_by` і таблиця `game_permissions`.

Рівні доступу:

- `VIEW` — користувач бачить гру і може керувати запусками до неї, але не редагує саму гру.
- `EDIT` — користувач може редагувати гру, завдання і сторінки, але не може видалити гру.

У редакторі гри метадані `Механіка`, `Умова перемоги` та `Автор` показуються в один рядок. Кнопка `Надати дозвіл на гру` розташована в цьому самому рядку. Сам блок керування дозволами прихований за замовчуванням і відкривається тільки після натискання кнопки.

Видалення гри доступне тільки автору гри або адміністратору.

### Сторінка `Користувачі і ролі`

На сторінці реалізовано:

- створення нового користувача;
- генерація логіна і пароля;
- копіювання даних входу в буфер обміну;
- список користувачів;
- живий пошук;
- зміна ролі;
- видалення користувача з підтвердженням паролем адміністратора.


### FEATURE: умовна поява контенту в межах завдання

Реалізовано механіку відкриття частини контенту після здачі коду.

STANDARD:
- умова задається на рівні секції конструктора контенту;
- варіанти: `відразу` або `після введення коду`;
- код перевіряється серед MAIN/BONUS/PENALTY кодів цього завдання;
- секція прихована для гравця, доки команда не здасть відповідний код.

MULTITASK:
- умова задається на рівні підзавдання;
- код перевіряється тільки серед кодів цього Multitask;
- підзавдання приховане, доки команда не здасть відповідний код.

OLYMPIAD:
- функція не застосовується.
